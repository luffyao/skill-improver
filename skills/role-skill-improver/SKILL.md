# Role Skill Improver

## 何时使用

在以下场景激活本 Skill：

- 某个角色 Skill 已可用，但触发不准、输出不稳定或成本偏高
- 你有 `evals` 结果、断言失败信息、或人工复盘反馈
- 需要形成迭代闭环，而不是一次性重写

## 输入

- 目标 Skill 路径
- 评测材料（至少一种）：
  - 断言结果
  - 执行轨迹摘要
  - 人工反馈
  - 与 baseline 的对比结果

## 按需读取参考资料

- 出现典型问题归类不清时，读取 `references/optimization-playbook.md`。
- 需要汇总多轮评测结果时，可运行 `scripts/aggregate_eval_stats.py`。

## 默认流程

1. 读取现有 Skill 与可用评测材料。
2. 必要时读取 `references/optimization-playbook.md` 选择优先改法。
3. 对问题做分层诊断（触发、流程、输出、稳定性）。
4. 按优化优先级生成改动方案。
5. 输出改进版 `SKILL.md` 与改动说明。
6. 给出下一轮 eval 建议。

## 优化优先级

按以下顺序优化，避免过度重构：

1. `description`（先修触发精度）
2. 流程指令（默认路径、步骤依赖、验收门槛）
3. `gotchas`（补常见误判与项目约束）
4. 输出模板（稳定结构化输出）
5. 脚本或参考资料（仅在重复劳动明显时引入）

## 必须遵守

- 先诊断后改动，避免直接重写全部内容
- 优先删除无价值指令，而非持续堆规则
- 改动必须说明“为何改、预期提升什么”
- 如果缺少足够证据，先请求补充输入

## 降级策略（输入不足时）

当输入材料不足时，按以下方式降级执行：

1. 若缺少 `eval` 和轨迹，仅基于 `SKILL.md` 做结构性优化建议，不直接大改。
2. 若只有零散反馈，优先收敛 `description` 与输出模板，不改复杂流程。
3. 若完全无证据，输出“待补充清单”，暂停实质性优化。

## 推荐输出结构

1. `Observed Issues`
2. `Root Causes`
3. `Proposed Changes`
4. `Updated SKILL.md`
5. `Expected Impact`
6. `Next Eval Plan`

## 轻量验证循环

1. 应用改动
2. 复跑最小 eval
3. 检查失败断言是否下降
4. 若无改善，回退高风险改动并重新收敛

## 质量检查清单

- [ ] 是否先诊断后改动
- [ ] 是否说明每个改动的原因与预期收益
- [ ] 是否给出下一轮可执行 eval 计划
- [ ] 输入不足时是否启用降级策略
