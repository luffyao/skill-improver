# 角色 Skill 系统实现计划

日期：2026-04-09  
关联设计：`docs/superpowers/specs/2026-04-09-role-skill-system-design.md`

## 目标

在现有四个 Skill 初版基础上，建立“可生成、可评测、可迭代优化”的稳定工作流，优先提升多仓库场景可用性与输出一致性。

## 里程碑

### M1：规范收敛（高优先）

**范围**

1. 统一四个 Skill 的输出章节命名（避免同义不同名）。
2. 为 `role-skill-creator` 增加“多仓库输入参数示例”。
3. 为 `role-skill-improver` 增加“最小输入不足时的降级策略”说明。

**验收**

- 四个 `SKILL.md` 的流程段落结构一致（触发、输入、流程、输出、禁区/校验）。
- Creator 示例中明确 `target_repo_paths` 使用方式。

---

### M2：评测加强（高优先）

**范围**

1. 扩充每个 Skill 的 eval 用例到 4-6 条。
2. 为断言增加可验证性（减少主观描述）。
3. 新增一个“多仓库冲突场景”专用 eval 集。

**验收**

- 每个 `evals/evals.json` 至少包含 1 条边界场景用例。
- 所有断言可明确判定 PASS/FAIL。

---

### M3：参考资料分层（中优先）

**范围**

1. 为 Creator 添加 `references/`：
  - `input-schema.md`（输入字段和约束）
  - `conflict-report-template.md`（冲突报告模板）
2. 为 Improver 添加 `references/`：
  - `optimization-playbook.md`（常见问题与改法）

**验收**

- `SKILL.md` 中明确“何时读取哪个 reference 文件”。
- 主文件保持精简，不堆叠过多细节。

---

### M4：脚本化辅助（中优先）

**范围**

1. 新增轻量脚本用于生成冲突报告骨架（可选）。
2. 新增 eval 统计脚本（聚合通过率、失败断言分布）。

**验收**

- 脚本支持 `--help` 且无交互输入。
- 输出为结构化格式（JSON 优先）。

---

### M5：稳定性复盘（持续）

**范围**

1. 运行至少一轮“with-skill vs baseline”对照。
2. 记录失败模式并回灌到 gotchas 或流程中。
3. 删除冗余指令，防止 Skill 变得臃肿。

**验收**

- 能给出至少 3 条“改动 -> 指标变化”的证据链。
- 触发误判率与无效步骤数量有可见下降。

## 执行顺序建议

`M1 -> M2 -> M3 -> M4 -> M5`

说明：先稳住结构和评测，再做脚本化，避免早期过度工程化。

## 风险与应对

1. **多仓库差异过大**：先报告冲突，不强行自动合并。
2. **断言不可判定**：把“好/坏”改成可观测标准。
3. **说明过长**：持续下沉细节到 `references/`。

## 完成定义（DoD）

满足以下条件视为该阶段完成：

1. 四个 Skill 均可独立被激活并输出稳定结构结果。
2. 多仓库冲突能被显式识别并提示用户决策。
3. 至少一轮评测迭代完成并形成改进记录。
