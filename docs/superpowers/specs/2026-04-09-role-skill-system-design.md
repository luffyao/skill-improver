# 多仓库角色 Skill 系统设计

日期：2026-04-09
状态：已确认（用户审批通过）

## 1. 目标

在 `skill-improver` 仓库中实现一套可复用的 Skill 体系，用于：

1. 为任意外部项目（支持多个仓库）生成角色 Skill。
2. 持续优化已生成的角色 Skill，使触发更准确、执行更稳定、输出更可评估。

本设计聚焦语言无关的软件工程场景，默认深度模式为自适应（`adaptive`）。

## 2. 范围与产物

本次交付四个 Skill：

1. `role-skill-creator`：生成角色 Skill（面向任意项目，支持多仓库）。
2. `role-skill-improver`：优化已有角色 Skill（基于评测与反馈迭代）。
3. `project-architect-role-template`：预置架构师角色模板。
4. `senior-software-developer-role-template`：预置高级开发者角色模板。

每个 Skill 至少包含：

- `SKILL.md`（主规范与流程）
- `evals/evals.json`（最小评测样例）

## 3. 关键需求

### 3.1 多仓库输入

`role-skill-creator` 需要支持：

- `target_repo_paths`：多个仓库路径列表
- 可选 `repo_aliases`：仓库别名
- 可选 `repo_priority`：优先级列表

### 3.2 冲突处理策略

当多仓库信息冲突时，采用用户确认的策略 B：

- 默认不自动定夺
- 先输出冲突清单、影响面、候选方案
- 请求用户确认后再收敛到单一决策

### 3.3 自适应深度

默认标准深度，命中高风险条件自动升级为深度模式。高风险包括：

- 核心架构改造、跨边界改动
- 安全、数据一致性、并发或发布风险
- 文档薄弱且代码复杂

升级后必须给出证据、风险分级、验证和回滚建议。

## 4. 架构设计

### 4.1 角色拆分

- `role-skill-creator`：负责“创建”
- `role-skill-improver`：负责“优化”
- 两个模板 Skill：提供可直接使用的角色基线

这种拆分避免单一 Skill 过度膨胀，符合 Agent Skills 的“职责内聚、可组合”原则。

### 4.2 渐进披露

`SKILL.md` 保持精简，包含核心触发条件、默认流程、验证环路和禁区。更长的内容（若后续需要）放入 `references/` 或 `assets/`，按需读取。

### 4.3 评测闭环

每个 Skill 提供最小 eval 示例，支持后续扩展为：

1. with-skill 与 baseline 对照
2. 断言评分
3. 人审反馈
4. 迭代更新

## 5. 数据流

### 5.1 创建流（Creator）

输入：多仓库路径 + 目标角色 + 深度模式 + 约束  
处理：仓库扫描 -> 事实提取 -> 冲突检测 -> 角色 Skill 生成 -> 最小 eval 输出  
输出：新角色 Skill 目录

### 5.2 优化流（Improver）

输入：目标 Skill + eval 结果 + 人工反馈 + 执行轨迹  
处理：问题诊断 -> 优先级修订（描述、流程、gotchas、验证） -> 产出改进版  
输出：修订后的 Skill 与改进说明

## 6. 输出格式要求

统一要求输出包含：

- 证据来源（尤其多仓库场景要标注仓库）
- 风险与假设
- 验证路径

架构师模板偏向方案与演进；高级开发模板偏向实现与回归控制。

## 7. 风险与缓解

1. **触发误激活**：通过更精确的 `description` 与 eval 迭代降低。
2. **多仓库冲突过多**：采用“冲突先提示”并要求用户确认。
3. **说明过重导致上下文压力**：保持 `SKILL.md` 精简，详细资料外置。

## 8. 验收标准

1. 四个 Skill 均符合 `SKILL.md` 基本规范（name/description/body）。
2. Creator 明确支持多仓库输入与冲突提示策略。
3. 两个角色模板可直接用于新项目角色化分析与执行。
4. Improver 可基于 eval/反馈提出结构化改进。

## 9. 非目标

1. 本次不实现重量级自动化脚本框架。
2. 本次不绑定特定语言或特定框架。
3. 本次不做 GUI 或 Web 面板。
